## Description

Implement `CoinbaseProvider` in `Sources/StockChartsKitCoinbase/` (PRD §9.2) against the Advanced Trade API (`/api/v3/brokerage`).

## Scope

- Authentication: **OAuth 2.0 with PKCE and no client secret** — only a client ID is stored (§5.1). Scopes: `wallet:accounts:read`, `wallet:transactions:read`. Drive via `AuthChallenge.oauthRedirect`.
- `listAccounts` → `GET /api/v3/brokerage/accounts`.
- `positions` → map each non-zero account to a `Position` with `assetClass = .crypto`, symbol like `BTC-USD`.
- `quote` → `GET /api/v3/brokerage/products/{product_id}/ticker`.
- `priceHistory` → `GET /api/v3/brokerage/products/{product_id}/candles`, granularity mapped from `TimeRange.granularity`.
- Capabilities: `[.positions, .balances, .quotes, .nativePriceHistory, .crypto]`.
- Use the shared `HTTPClient` (#0014); map errors to `BrokerageError`.

## Acceptance criteria

- HTTP replay tests (#0024) in `StockChartsKitCoinbaseTests` for accounts, positions, ticker, candles. No live tokens committed.
- Client ID supplied via `Configuration` from Keychain — no secret in source/fixtures.
- `swift build` + `swift test` pass with no live network.

PRD §9.2 (Coinbase), §5.1 (PKCE, no client secret).

New implementation, no prior bug.

## Fix

Implemented `CoinbaseProvider` as an `actor` conforming to `BrokerageProvider`
in `Sources/StockChartsKitCoinbase/`, replacing the placeholder enum.

- **Auth — OAuth 2.0 with PKCE, no client secret.** Only `clientID` and
  `redirectURI` are read from `Configuration`; there is no `clientSecret`.
  `authenticate()` vends a `CoinbaseAuthSession` holding a fresh PKCE
  verifier/challenge pair and an opaque `state`. The authorization URL carries
  `response_type=code`, `client_id`, `redirect_uri`, `scope`
  (`wallet:accounts:read wallet:transactions:read`), `state`, `code_challenge`,
  and `code_challenge_method=S256`. `complete(callbackURL:)` validates `state`,
  extracts `code`, and exchanges it (with the verifier) at
  `https://login.coinbase.com/oauth2/token` for access/refresh tokens, which are
  persisted through an injected `TokenStore` as a JSON `TokenBundle`.
  `authenticationStatus()` returns `.connected` when a valid/refreshable token
  exists, else `.requiresChallenge(.oauthRedirect(...))`.
- **Token storage/refresh.** A new `TokenStore` protocol abstracts persistence;
  the package's `KeychainStore` conforms to it for production, and tests inject
  an in-memory store (the Keychain is unavailable under `swift test`). A
  `CoinbaseTokenProvider` actor vends a valid bearer token to the `HTTPClient`
  `RequestSigner` on every request, refreshing via `grant_type=refresh_token`
  when the access token is within 60s of expiry.
- **Reads** (Bearer-signed via the shared `HTTPClient`): `listAccounts` →
  `GET /api/v3/brokerage/accounts`; `positions` maps each non-zero-balance
  account to a `.crypto` `Position` with symbol `<CCY>-USD`; `balance` sums
  available + hold; `quote` → `.../products/{id}/ticker`; `priceHistory` →
  `.../products/{id}/candles` with `start`/`end` from `range.interval()` and a
  Coinbase `granularity` mapped from `TimeRange.granularity`.
- **Capabilities** `[.positions, .balances, .quotes, .nativePriceHistory,
  .crypto]`. SHA-256 via the system `CryptoKit` framework, so no package
  dependency was added to the target.

## Gotchas

- **SHA-256: CryptoKit vs swift-crypto.** Chose the system `CryptoKit`
  framework for PKCE so the target needs no extra package dependency. The
  `swift-crypto` `Crypto` module was the documented alternative but was avoided
  to keep the dependency graph minimal.
- **Granularity mapping.** `Granularity` → Coinbase enum:
  `oneMinute`→`ONE_MINUTE`, `fiveMinutes`→`FIVE_MINUTE`,
  `fifteenMinutes`→`FIFTEEN_MINUTE`, `oneHour`→`ONE_HOUR`, `oneDay`→`ONE_DAY`.
  Coinbase has no weekly candle, so `oneWeek` (used by 5Y/MAX) falls back to
  `ONE_DAY`; the chart layer downsamples.
- **Token storage/refresh.** A `TokenStore` protocol (conformed to by
  `KeychainStore`) was introduced because `KeychainStore` is a concrete actor
  and the Keychain is unavailable under `swift test`. Tokens live as one JSON
  `TokenBundle` keyed `coinbase.<connectionID>`; the signer refreshes them
  lazily via an actor (`CoinbaseTokenProvider`) so concurrent requests share a
  single refresh.
- **`HTTPClient` decoder snake_case.** The shared decoder applies
  `.convertFromSnakeCase`; the initial `TokenResponse` with explicit
  `access_token` coding keys failed to decode. Removing the custom keys (letting
  the strategy map `access_token`→`accessToken`) fixed it.
- **Auth tested offline.** PKCE math is asserted directly against an
  independently computed SHA-256; the authorization URL is built and inspected
  without any network; the token exchange is driven through a replayed
  `/oauth2/token` fixture and an in-memory token store, then
  `authenticationStatus()` is checked before/after.
