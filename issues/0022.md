## Description

Implement the small `SnapshotScheduler` helper (PRD §7) that the host app uses to call `recordSnapshot` on a cadence (e.g. hourly while foregrounded, daily otherwise). The package provides the helper but does **not** manage launch agents or background tasks — that's the app's job.

## Scope

- A lightweight scheduler that, given providers + a `SnapshotStore`, periodically records `PortfolioSnapshot`s.
- Async/await based (no Combine, no timers that capture self unsafely under strict concurrency).
- Configurable interval; cancellation-aware.
- Does not own OS scheduling — exposes start/stop the host drives.

## Acceptance criteria

- Tests with an injected clock / fast interval and `InMemorySnapshotStore` (#0015): scheduler records the expected number of snapshots and stops cleanly on cancel.
- `swift build` + `swift test` pass.

PRD §7 (SnapshotScheduler; app owns OS scheduling).

New implementation, no prior bug.

## Fix

Added `SnapshotScheduler` (`Sources/StockChartsKit/Snapshot/SnapshotScheduler.swift`),
a `Sendable`-safe `actor`:

- Constructed with `providers: [any BrokerageProvider]`, a `store: any SnapshotStore`,
  an `interval: Duration`, an injectable `sleep: @Sendable (Duration) async throws -> Void`
  (defaults to `Task.sleep(for:)`), and an injectable `now: @Sendable () -> Date`
  (defaults to `Date()`) for deterministic snapshot timestamps in tests.
- `recordOnce()` walks every provider, calls `listAccounts()`, and for each account
  fetches `balance(for:)` and `positions(for:)`, builds a
  `PortfolioSnapshot(accountID:timestamp:totalValue:positions:)` (totalValue from the
  balance's `total`), and calls `store.recordSnapshot(...)`. A single provider or
  account failure is logged via `os.Logger` (subsystem `co.sstools.stockchartskit`,
  category `SnapshotScheduler`, never logging secrets) and skipped; the rest of the
  pass continues. `recordOnce()` never throws.
- `start()` launches a stored `Task` running a loop: record a pass, sleep the
  interval, repeat until cancelled; `start()` is idempotent. `stop()` cancels the
  task, awaits its completion (`await task.value`), and clears it — so a running task
  can never leak. The loop checks `Task.isCancelled` between passes and breaks if the
  (cancellation-aware) sleep throws.
- Per PRD §7 the helper does **not** manage launch agents or background tasks; the
  host drives `start()`/`stop()`.

## Gotchas

- **Clock/sleep injection.** The sleep is `@Sendable (Duration) async throws -> Void`
  defaulting to `Task.sleep(for:)`; tests inject a `SleepGate` actor that suspends on a
  stored continuation and is resumed explicitly. The snapshot timestamp comes from an
  injected `now` closure (`Date()` in production) so test assertions are deterministic.
- **Partial-failure handling.** Failures are caught per-provider (`listAccounts`) and
  per-account (`balance`/`positions`/`recordSnapshot`), logged, and skipped;
  `recordOnce()` never throws so one bad provider/account can't abort the pass.
- **No leaked task / clean stop.** `start()` stores the `Task` and is idempotent;
  `stop()` cancels it and `await`s `task.value` before clearing. The injected sleep
  must be cancellation-aware (the real `Task.sleep` is) or `stop()` would hang — the
  test's `SleepGate` uses `withTaskCancellationHandler` to resume the pending sleeper
  with `CancellationError` on cancel, mirroring production behaviour and proving the
  loop terminates. The `stop()` test was wrapped in a task-group timeout so any
  regression that reintroduces a hang fails fast instead of blocking the suite.
