## Description

Implement the authentication primitives in `Sources/StockChartsKit/Auth/` (PRD §5, §5.1). These are shared across all providers: a challenge enum the host resolves, a session protocol the host drives, and a Keychain-backed credential store.

## Scope

- `AuthChallenge.swift`: `enum AuthChallenge: Sendable` with `oauthRedirect(authorizationURL:callbackScheme:)`, `pinCode(authorizationURL:)`, `mfaPushNotification(prompt:)`, `hostedPortal(url:)` (§5).
- `AuthSession.swift`: `protocol AuthSession: AnyObject, Sendable` with `complete(callbackURL:)`, `complete(pinCode:)`, `cancel()`.
- `KeychainStore.swift`: `actor KeychainStore` wrapping `SecItemAdd` / `SecItemCopyMatching` / update / delete. Per provider, store one item keyed `\(providerID).\(connectionID)` with service = app bundle identifier. Written directly against the Security framework — **no KeychainAccess dependency** (§13).
- Add `Configuration` struct (PRD §5.1): `secrets: [String: String]`, `init(secrets:)`. The package never reads secrets from env/disk on its own.

## Acceptance criteria

- **Never log token values** (§5.1, §18) — verify no logging of secret material.
- KeychainStore round-trip tests where feasible (may require a test keychain / entitlement; document if not runnable in CI sandbox).
- `swift build` passes.

## References

PRD §5 (authentication), §5.1 (secrets & configuration), §13 (no KeychainAccess).

## Root cause

New implementation, no prior bug.

## Fix

Added the auth layer in `Sources/StockChartsKit/Auth/`, in the core target with
no external dependencies:

- `AuthChallenge.swift` — `enum AuthChallenge: Sendable` with the four cases
  (`oauthRedirect`, `pinCode`, `mfaPushNotification`, `hostedPortal`).
- `AuthSession.swift` — `protocol AuthSession: AnyObject, Sendable` with
  `complete(callbackURL:)`, `complete(pinCode:)`, `cancel()`.
- `Configuration.swift` — `struct Configuration: Sendable` carrying
  `secrets: [String: String]`, plus a `requiredSecret(_:)` helper that throws
  `BrokerageError.authenticationFailed` (naming the key, never the value) on a
  missing key. The package reads no secrets from env/disk on its own.
- `KeychainStore.swift` — `actor KeychainStore` wrapping `SecItemAdd` /
  `SecItemCopyMatching` / `SecItemUpdate` / `SecItemDelete` directly (no
  KeychainAccess dependency). One generic-password item per key with
  `kSecAttrService` = the service string; `secret(_:)`, `set(_:for:)`,
  `delete(_:)`. Failures map to `BrokerageError`; secret values are never
  logged (only operation names and `OSStatus` codes, at `.public` privacy).

## Verification

- `swift build` — Build complete, no warnings.
- `swift test` — Test run with 35 tests in 9 suites passed (was 26; +9 added
  across Capabilities, Configuration, AuthenticationStatus, AuthChallenge, and
  KeychainStore suites).
- KeychainStore round-trip (`set` → `secret` → `delete` → expect-throws on
  re-read): this test ran and passed in this sandbox (~0.022s) — the Keychain
  was available, so the round-trip was fully exercised. The test still contains
  a skip path: if `SecItem*` returns an entitlement/availability status
  (`errSecMissingEntitlement`, `errSecNotAvailable`, etc.) it is treated as a
  skip rather than a failure, so the suite stays green on a sandbox that lacks
  Keychain access. No pass is faked.
