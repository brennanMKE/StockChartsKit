## Description

`Package.swift` currently declares only the core `StockChartsKit` target and its test target. It must declare every target and product the PRD calls for (PRD §2, §13), one product per provider so the host app can pick which to link, with strict concurrency enabled and pinned dependencies.

## Scope

- `swift-tools-version:6.1` (or later already in repo), platforms `.macOS(.v14)`.
- Enable strict concurrency (`SwiftSettings.enableUpcomingFeature("StrictConcurrency")`).
- Targets: `StockChartsKit` (core, only depends on `swift-crypto` + `swift-collections`), `StockChartsKitTesting`, `StockChartsKitMarketData`, `StockChartsKitCSV`, `StockChartsKitCoinbase`, `StockChartsKitETrade`, `StockChartsKitSchwab`, `StockChartsKitSnapTrade`.
- One library product per provider + core + market data + testing.
- Matching test targets per PRD §2: `StockChartsKitTests`, `StockChartsKitETradeTests`, `StockChartsKitCoinbaseTests`, `StockChartsKitSnapTradeTests`, `StockChartsKitSchwabTests`, `StockChartsKitCSVTests`.
- Dependencies (pinned, PRD §13): `apple/swift-crypto`, `groue/GRDB.swift`, `apple/swift-collections`, `swift-testing` (toolchain-provided). No Alamofire, KeychainAccess, SwiftJWT, Vapor.
- Create the empty source directory skeleton from PRD §2 so the targets resolve.

## Acceptance criteria

- `swift build` resolves all targets (empty/stub sources are fine at this stage).
- Core target has no provider-specific or third-party deps beyond `swift-crypto` + `swift-collections`.

## References

PRD §2 (package layout), §13 (dependencies), §1 (platform requirements).

## Root cause

New implementation, no prior bug. `Package.swift` only declared the core
`StockChartsKit` target/product and its test target; the rest of the PRD §2
skeleton was missing.

## Fix

- Declared 7 new targets alongside core: `StockChartsKitTesting`,
  `StockChartsKitMarketData`, `StockChartsKitCSV`, `StockChartsKitCoinbase`,
  `StockChartsKitETrade`, `StockChartsKitSchwab`, `StockChartsKitSnapTrade`.
  Each new target got a tiny stub source (doc comment + empty `enum`
  namespace) so SPM finds sources.
- Declared one library product per provider plus `StockChartsKit`,
  `StockChartsKitMarketData`, `StockChartsKitTesting`.
- Added 5 new test targets (`...ETradeTests`, `...CoinbaseTests`,
  `...SnapTradeTests`, `...SchwabTests`, `...CSVTests`), each with a stub
  importing `Testing`, its source target, and `StockChartsKitTesting`.
- Pinned dependencies: `apple/swift-crypto` (`from: 3.0.0`),
  `groue/GRDB.swift` (`from: 6.0.0`),
  `apple/swift-collections` (`.upToNextMajor(from: 1.1.0)`).
  `swift-testing` is toolchain-provided and not added as a package dep.
- Wiring: every provider target + MarketData + Testing depend on
  `StockChartsKit`. `Crypto` added to `StockChartsKitETrade` and
  `StockChartsKitSnapTrade` only. `GRDB` + `Collections` added to core.
- Strict concurrency applied consistently via a shared `swiftSettings`
  array (`enableUpcomingFeature("StrictConcurrency")`) plus the existing
  `swiftLanguageModes: [.v6]`. MainActor default isolation was NOT
  reintroduced.

## Verification

## Gotchas

- **PRD §2 dependency-wording inconsistency.** The prose at the end of §2
  says the core target has "no third-party dependencies beyond swift-crypto
  and the standard library", yet the same §2 layout places
  `SQLiteSnapshotStore` (GRDB) and time-series code (swift-collections) in
  the core target, and §13 lists GRDB + swift-collections as deps. Resolved
  pragmatically: the core `StockChartsKit` target depends on `GRDB` and
  `Collections`. Notably the core target does NOT depend on `swift-crypto` —
  crypto is only used by the ETrade and SnapTrade signers, so `Crypto` is
  scoped to those two targets only.
- Which targets got which deps: core -> `GRDB` + `Collections`;
  `StockChartsKitETrade` + `StockChartsKitSnapTrade` -> `Crypto`; all other
  provider/market-data/testing targets -> just `StockChartsKit`.
- `swiftLanguageModes: [.v6]` already enforces strict concurrency; the
  `enableUpcomingFeature("StrictConcurrency")` flag is redundant under v6
  mode but applied per PRD §1 for explicitness and forward-compat. MainActor
  default isolation was intentionally left off (core types are nonisolated
  Sendable value types).
