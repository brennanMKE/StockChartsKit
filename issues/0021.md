## Description

Implement the SnapTrade integration in `Sources/StockChartsKitSnapTrade/` (PRD §9.5) — the workhorse covering Robinhood, Fidelity (retail + NetBenefits), Acorns, Vanguard, Wealthfront, Webull, Public, Wells Fargo, etc. Exposes **one provider per connected brokerage**, not one global provider.

## Scope

- `Signing.swift`: SHA-256 HMAC signature in the `Signature` header on every request, using swift-crypto.
- Authentication flow:
  1. Register the package's `clientId` + `consumerKey` with SnapTrade (from `Configuration`/Keychain).
  2. `POST /api/v1/snapTrade/registerUser`.
  3. `POST /api/v1/snapTrade/login` → present hosted portal via `AuthChallenge.hostedPortal`.
  4. Persist `userId` + `userSecret` in Keychain.
- `SnapTradeProviders.discover(configuration:snapshotStore:marketData:)` → returns `[any BrokerageProvider]`, one per registered `Connection`.
- Provider IDs namespaced: `snaptrade.robinhood`, `snaptrade.fidelity`, `snaptrade.acorns`, … — brokerage name from the `Account.brokerage_authorization` field.
- Endpoints under `https://api.snaptrade.com/api/v1`: `GET /accounts`, `/accounts/{id}/positions`, `/accounts/{id}/balances`, `/accounts/{id}/activities` (for snapshot reconstruction).
- Capabilities: `[.positions, .balances]`. No native price history — rely on `MarketDataProvider`.
- **Fidelity/NetBenefits**: enumerate whatever SnapTrade returns; do not assume a single login sees both retail and workplace accounts.

## Acceptance criteria

- Unit tests for the SHA-256 HMAC signer against known vectors.
- HTTP replay tests for registration, login URL generation, accounts/positions/balances, and `discover()` yielding multiple namespaced providers.
- `clientId`/`consumerKey`/`userId`/`userSecret` never hardcoded or logged.
- `swift build` + `swift test` pass with no live network.

## Fix

Implemented the full SnapTrade integration in `Sources/StockChartsKitSnapTrade/`:

- **`Signing.swift`** — `SnapTradeSigner`, HMAC-SHA256 via `import Crypto`, typed
  `throws(SnapTradeSigningError)`. Signs the canonical signature object
  `{"content":<body|null>,"path":"<path>","query":"<sorted-encoded-query>"}`,
  serialized with `JSONSerialization` `.sortedKeys`/`.withoutEscapingSlashes` for
  byte determinism, keyed by the `consumerKey` (UTF-8), base64-encoded into the
  `Signature` header. Query canonicalization sorts by name (ties by value) and
  percent-encodes per RFC 3986 unreserved set. The timestamp is injectable.
- **`SnapTradeAPI.swift`** — builds signed `URLRequest`s, attaching `clientId`,
  `userId`, `userSecret`, `timestamp` as query params and the `Signature` header,
  reusing the exact canonical query string on the wire so it matches what was
  signed. Also hosts `ProviderID.snapTrade(brokerageName:)` slugging.
- **`SnapTradeConnectionProvider.swift`** — an `actor` `BrokerageProvider`, one
  per connected brokerage, namespaced id `snaptrade.<slug>`, capabilities
  `[.positions, .balances]`. `quote`/`priceHistory` throw `.unsupported`. Reads
  `GET /accounts` (filtered to its `brokerage_authorization`),
  `/accounts/{id}/positions`, `/accounts/{id}/balances`.
- **`SnapTradeProviders.swift`** — `registerUser` (`POST /snapTrade/registerUser`,
  persists `userId`/`userSecret`), `loginURL`/`loginChallenge`
  (`POST /snapTrade/login` → `AuthChallenge.hostedPortal`), and `discover(...)`
  which groups `/accounts` by `brokerage_authorization` and returns one provider
  per connection.
- **`TokenStore.swift`** — narrow credential-store protocol (Keychain in prod,
  in-memory in tests) plus `SnapTradeUserCredentials` codable storage. Never
  logged.
- **`SnapTradeWireTypes.swift`** — decode-only wire types using `LossyDecimal`
  for money/quantity (string-or-number JSON), with a `brokerageAuthorization`
  decoder tolerating both embedded-object and bare-string shapes.

## Verification

- `swift build` — clean.
- `swift test` — 149 tests pass (was 135; +14 SnapTrade), 24 suites, offline
  (HTTP replay via `ReplayURLProtocol`, in-memory `TokenStore`; no live network,
  no real credentials). Replay hit counts asserted (e.g. `/accounts` = 1,
  `registerUser` = 1, `login` = 1).
- **Signer validation**: there is **no published official SnapTrade HMAC test
  vector**, so the canonical server contract cannot be vector-checked offline.
  Instead the signer is validated by (a) determinism (identical inputs → identical
  signature; differing input → differing signature), (b) an exact-bytes assertion
  on the canonical signature-object JSON, and (c) equality against an
  independently computed `HMAC-SHA256` (base64) over those exact bytes with a
  synthetic key — proving the HMAC wiring is correct.
- **discover()**: an `/accounts` fixture spanning two connections (Robinhood +
  Fidelity, the latter with 2 accounts incl. a NetBenefits 401k) yields exactly
  two providers with ids `snaptrade.robinhood` / `snaptrade.fidelity` and the
  correct display names.
- **No secrets**: source and fixtures grepped — only synthetic/REDACTED values
  (`synthetic-client-id`, `synthetic-consumer-key`, `REDACTED-SYNTHETIC-USER-SECRET`,
  `SYNTHETIC-CONNECT-TOKEN`). No `print()`; logging via `os.Logger` category
  `snaptrade`.

## Gotchas

- **Exact bytes signed / no official vector**: the signature object is
  `{"content":<parsed body or null>,"path":"/api/v1/...","query":"<sorted-encoded>"}`
  serialized with sorted keys and no insignificant whitespace; the `query` is the
  same canonical string put on the wire. There is **no published official SnapTrade
  HMAC vector**, so offline validation proves HMAC correctness, not the live server
  contract — confirm against a sandbox call before shipping live.
- **Namespaced-id slugging**: brokerage name → lowercased, non-alphanumeric runs
  collapse to a single `-`, trimmed; `"Fidelity NetBenefits"` →
  `snaptrade.fidelity-netbenefits`, `"Wells Fargo"` → `snaptrade.wells-fargo`.
- **Fidelity/NetBenefits enumeration**: `discover()` makes no assumption about a
  single login's account set — it groups by `brokerage_authorization` and emits
  one provider per authorization, so multiple accounts under one Fidelity login
  collapse into one `snaptrade.fidelity` provider, and a separate NetBenefits
  login (distinct authorization) would surface as its own provider.
- **userId/userSecret storage**: stored as JSON under the single Keychain key
  `snaptrade.user` (credentials are shared across all SnapTrade connections);
  `signOut()` on any connection clears them for all. Never logged.
- **brokerageAuthorization decoding** tolerates both an embedded object (with the
  brokerage name) and a bare id string; a bare-string authorization yields a
  provider named after its id since no human name is available.
