## Description

Implement the SnapTrade integration in `Sources/StockChartsKitSnapTrade/` (PRD §9.5) — the workhorse covering Robinhood, Fidelity (retail + NetBenefits), Acorns, Vanguard, Wealthfront, Webull, Public, Wells Fargo, etc. Exposes **one provider per connected brokerage**, not one global provider.

## Scope

- `Signing.swift`: SHA-256 HMAC signature in the `Signature` header on every request, using swift-crypto.
- Authentication flow:
  1. Register the package's `clientId` + `consumerKey` with SnapTrade (from `Configuration`/Keychain).
  2. `POST /api/v1/snapTrade/registerUser`.
  3. `POST /api/v1/snapTrade/login` → present hosted portal via `AuthChallenge.hostedPortal`.
  4. Persist `userId` + `userSecret` in Keychain.
- `SnapTradeProviders.discover(configuration:snapshotStore:marketData:)` → returns `[any BrokerageProvider]`, one per registered `Connection`.
- Provider IDs namespaced: `snaptrade.robinhood`, `snaptrade.fidelity`, `snaptrade.acorns`, … — brokerage name from the `Account.brokerage_authorization` field.
- Endpoints under `https://api.snaptrade.com/api/v1`: `GET /accounts`, `/accounts/{id}/positions`, `/accounts/{id}/balances`, `/accounts/{id}/activities` (for snapshot reconstruction).
- Capabilities: `[.positions, .balances]`. No native price history — rely on `MarketDataProvider`.
- **Fidelity/NetBenefits**: enumerate whatever SnapTrade returns; do not assume a single login sees both retail and workplace accounts.

## Acceptance criteria

- Unit tests for the SHA-256 HMAC signer against known vectors.
- HTTP replay tests for registration, login URL generation, accounts/positions/balances, and `discover()` yielding multiple namespaced providers.
- `clientId`/`consumerKey`/`userId`/`userSecret` never hardcoded or logged.
- `swift build` + `swift test` pass with no live network.
